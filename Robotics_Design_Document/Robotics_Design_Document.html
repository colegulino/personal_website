Executive Summary

Each year, the Institute of Electrical and Electronics Engineers (IEEE) Association Region 5 hosts a competition which pits students within the region against each other in a competition to see which group of students can design a robot that can perform the given task under certain constraints. Each year the competition changes giving students new and exciting challenges but retains the same theme of the designed robot being able to take information from its surroundings and autonomously make decisions as to its next course of action.

Louisiana State University has competed in this competition for a number of years, placing in a few of them. This year the team plans to do the same and win the competition gaining reputation for the school and grow its ever increasing list of academic accomplishments. The competition this year places the robot in a maze where it is expected to find its way out and determine the shortest or “critical” path. The maze will be randomized and increase in size each round culminating in a 7’ by 7’ maze. Upon the walls of the maze will be characters that, if recognized by the robot, can add to the final score. The robot will have a set time limit in which to solve the maze and scan the walls which will lead into the second part of the round where the robot will attempt the critical path without deviation.

The team researched many different options for our platform but ultimately decided that treads would best suit this competition given its ruggedness and its ease to overcome obstacles. By searching online, the team found a small tread kit that can easily be implemented in our design. These treads are cheap, provide good traction, fit within the size constraint, and would provide the means to overcome speed bumps present in the maze. The next step was to find a motor that would be propel our robot through the maze. After extensive research, the team decided to use the Pololu 150:1 Micro Metal Gearmotor since it will provide the needed torque and a quick speed. It also allows each motor to act individually so that our robot can turn.

In order to take in the various inputs, control the motors, and process the data necessary, a powerful controller and processor must be used in order to ensure that the robot can navigate the maze successfully.  In order to accomplish this, we have chosen the Raspberry Pi B+ as our controller.  The Raspberry Pi B+ has a fast processor for its size and speed.  The Raspberry Pi B+ also has numerous GPIO ports and a dedicated camera port.  This allows it to meet all of our I/O needs.  In order to run our robot, the team also needed to choose a programming language compatible with the Raspberry Pi B+ that has enough packages to run the functions necessary.  The Python programming language was chosen.  Python is a flexible and readable language that makes it great for collaboration.  It is also compatible with numerous packages necessary to control the robot such as: OpenCV, RPi.GPIO, and PiCamera.  The Raspberry Pi B+ running Python will meet all of our controller needs.

In order to maneuver the robot around the maze, it must be able to detect its surroundings. For this, the team developed a sensor package consisting of 3 infrared proximity sensors to detect the walls, and an array of 5 reflectance sensors to detect the lines bisecting each square of the maze. One issue with the sensors selected is that each sensor outputs analog information, and the Raspberry Pi controller has no onboard analog to digital converters. A circuit board using a microcontroller with onboard ADCs was designed in order to read each sensor and relay binary information about the walls and ground to the Raspberry Pi.

The robot requires vision capabilities to successfully capture, recognize and save characters on the maze walls. After comparing many alternatives in both hardware and software for the vision system, in categories such as power, speed, and overall cost, the team chose to use the Raspberry Pi Camera Module to implement the robot’s vision system. The Raspberry Pi Camera Module is comparatively cheap, fast, and has a dedicated connection port to the Raspberry Pi B+. On the software side, the team chose to use the Python implementation of OpenCV to cover any image processing.

In order to run the various components mentioned above, the robot requires a robust and stable power system.  After researching many batter technologies, the team chose rechargeable NiMH technology.  The team designed the power system of the robot in such a way that we can use the single NiMH battery for every component by cascading the voltage level down in steps with regulators. The team created simple +12V and +5V rails for all our components to run from which in turn helped keep complexity down.  We also designed a simple detection circuit for the battery that allowed us to see when the voltage had dropped too low below nominal voltage and needed to be charged.

All of these components will be housed on the chassis which we will develop in-house using the resources available in the machine shop. This chassis will be approximately 7 in. long and 5 in. in width. Each piece will be attached with either bolts or screws to ensure each piece will be securely placed.

In 2013 robotic sales increased 12% over 2012, which is the highest number of robots ever sold.  This growth shows that this industry will require educated young engineers.  Each year Region 5 of IEEE holds a conference, which features a robotics competition between student teams in the region.  A fully autonomous robot is needed to compete in this competition for LSU.  The robot must be able to take in sensory data from the maze, dynamically determine a critical path for solving the maze, and relay maze information to the judges in the form of ASCII character “easter eggs” found on the maze walls.  The robot will be given an amount of time to first explore the maze and map it, then a smaller amount of time to make a “critical run path” from the end point back to the start without deviation.

Background

This year’s IEEE R5 Robotics Competition is a maze solving challenge.  The team’s task is to build a robot that can autonomously navigate, extract information, and solve a maze in under a specified amount of time.  The competition is composed of 3 qualifying rounds and a final round.  The competition maze will come in three sizes 5’x5’, 6’x6’, and 7’x7’.  Each successive qualifying round the maze will increase in size, and the final round’s maze will have a size of 7’x7’.  Each maze will have a specified starting and ending point, and the rest of the maze will be randomly generated.  There will only be one critical path that will solve the maze.  Each round is further broken into two parts: the first part being where the robot explores the maze and learns about it, and the second being when the robot attempts to solve the maze by running the critical path.  In order for the robot to be able to compete in the final round, it must qualify by completing any critical path run from one of the three qualifying rounds.

The first part of each round is primarily focused on exploration and discovery of the maze itself.  Points are awarded for this round by moving into new spaces and moving along the critical path, while losing points for repeating moves and bumping into the walls.

Scoring

Here is a detailed breakdown of the scoring system for the qualifying rounds:

  • Entering New Square (Part 1) +1
  • Correct Critical Path Move +2
  • Lighting A Correct Light +1
  • Repeating A Path Move -1
  • Touching Wall (per square) -1
  • Moving Off Critical Path (part 2) -1
  • Easter Egg Number +1
  • Easter Egg Alphabetical +2
  • Easter Egg Symbols NA

For the final round, the winner is decided by a total adjusted time for the critical path run during the final round.  Runtime is lowered for each character identified during the final round of the competition.  In addition to the qualifying round’s system of scoring, the final round also adds extra scoring opportunities:

  • Adjusted Critical path Execution Time – Winning Criteria
  • Search (part 1) Execution Time – Tiebreaker
  • BTB – Bonus Time Base – Fastest Critical Path Time in Finals
  • Easter Egg – Number - +1 plus 5% of BTB off critical path time
  • Easter Egg – Letter - +2 plus 15% of BTB off critical path time
  • Easter Egg – Other – No score

By identifying Easter eggs on the walls of the maze during the final round, the team can lower its final runtime.  However this bonus time we can eliminate from our run time is capped at 35% of BTB.

Rules

Along with scoring rules, there are also special rules in place for the competition.

  • No robot kits allowed, only original designs
  • Damaging walls or maze will result in disqualification
  • Robots can never extend above 7.5” tall throughout the competition
  • Robots cannot split into separate parts, as well as cannot fly, jump, or climb over walls
  • Robots will be sequestered between rounds by the judges and powered down
  • Robots that stop moving for more than 10 seconds will have their current round ended
  • Robots must have an easily accessible start and stop button
  • No wireless communication is allowed with the robot during the competition rounds

Processing

A controller/processor must control the movement of a robot, process information from sensors, including a camera.  A processor must be able to process these simultaneously while having enough processing power to execute these quickly enough to move, map the maze, process and analyze images, and determine a critical path under specified time constraints.  The processor must also be able to many inputs and outputs.  

A microcontroller is a piece of hardware that has a CPU and a fixed amount of memory: RAM and/or ROM.  Certain microcontrollers, such as the Arduino, have other hardware devices built in such as analog to digital converters.  Certain boards are also built with connections like dedicated motor drivers, dedicated camera ports, and other hardware meant to facilitate specific tasks.  A microcontroller is made for a specific purpose: controlling motors, microwaves, cars, washing machines, etc.  The CPU of a microcontroller has a specific type of architecture.  Some examples are: ARM, Intel x86, MIPS, etc.  The boards that a microcontroller is built with has a specific design made by the company that produces the controller.  For example, the Arduino microcontroller can have an Intel or ARM based CPU architecture, but the board was made and designed by the company that produces the board.  The computer architecture firm will usually source out their CPU architectures to multiple different board makers.  Architectures are built around the number of bits that their instructions contain.  Most architectures run on 32-bit instructions, but recently 64-bit instructions have become vogue.  This allows for unification of software designs across multiple platforms.  This allows compilers to run on multiple devices without having to be specifically built for certain computer architectures.  

Certain microcontroller producers will design proprietary software that will only be able to run on their devices.  This software can facilitate certain functions like accessing board pins and analog to digital converters.  Other boards which are not programmed with proprietary programming languages are able to run multiple types of software like: C++, Java, Python, etc.  These non-proprietary programming languages can have libraries built for them that run similar features on other microcontrollers that a proprietary programming language would run.  A controller that can run multiple programming languages other than just proprietary software allows for flexibility in the design of software systems.  

Microcontrollers communicate with a computer in multiple ways.  This hardware is designed to either communicate in serial or in parallel with the computer.  Some ways that the microcontroller can communicate with the computer is through: UART, USART, or TTL serial communication.  UART stands for universal asynchronous receiver/transmitter.  It is usually part of the integrated circuit and part of the computer architecture design.  The form that the data can take is variable and controlled by a driver circuit which is external the UART, probably on the IC.  The UART takes bytes of data and transmits them serially to the computer.  A second UART reassembles the bits into full bytes which can be read and understood by the computer.  Each UART contains a shift register and one wire for communication.  Serial transfer is less costly than parallel communication requiring multiple wires.  USART stands for universal synchronous/asynchronous receiver/transmitter.  The USART is similar to the UART, but it offers the option of a synchronous mode, which requires that each end of an exchange respond in turn without initiating a new communication.  Synchronous mode also requires a clock.  Serial mode also allows for a higher data transfer rate than asynchronous mode.  Finally TTL serial communication is a form of serial communication (UART or USART) which transfers one bit at a time.  This communication is based on 0s being ground and 1s being Vcc.  This Vcc is based on the power supply of the controller.

arduino.cc

The Arduino DUE is a microcontroller is based on the ARM CPU architecture.  It can communicate through the computer through one UART, two USART.  The TTL communication is based on 3.3V.  The Arduino DUE is programmed with proprietary software.  This software is based on C++ and has a large development community which builds libraries to support pin I/O, servo motors, stepper motors, cameras, LCD screens, etc.  The Arduino DUE has two USB ports: one for programming and one native port.  The native port runs a “soft erase” in order to reset the programming on the Arduino board.

Another class of microcontrollers are those that run full computer operating systems such as: Linux, Unix, Windows Embedded, etc.  These microcontrollers have the full suite of programs as regular computers, but their architecture is made for a smaller embedded system that is made to run more complicated software than a simple microcontroller, but is made to be run on a specific and limited hardware in comparison to a full computer. An operating system is the low level software of a computer or microcontroller that operates as an interface between the hardware of the computer and the high level software from applications or computer operators.  Operating systems like Windows and OSX, which is a form of Unix, are highly developed software systems that are made by large corporations. The Linux operating system was designed as an open-source version of the Unix operating system.  Linux has been developed collaboratively since the 90s undergoing many iterations.  There are numerous variations of Linux such as: Ubuntu, Debian, RHEL (Red Hat Enterprise Linux).  There are also many operating systems that are repurposed versions of Linux.  Some of the main examples are Google’s Mobile System, Android, Google’s Desktop System, Chrome OS, and Raspbian which is a repurposed version of Linux Debian designed to run on the Raspberry Pi.  Microcontrollers with full-fledged operating systems are sometimes termed “microcomputers”.

Propeller C3

parallax.com

The Propeller C3 is a “Credit-Card-sized-Computer” made by Parallax Inc. which is designed as an add-on to the standard Propeller CPU which allows it to function as a microcomputer beyond what a regular microcontroller can do.  The Propeller C3 has an SPI (Serial Peripheral Interface) which allows for serial sequential data.  This is an industry standard interface which allows the Propeller C3 to work with many different types of sensors and interfacing apparatus.  The Propeller C3 also comes with a 2 channel SPI analog to digital converter.  It also comes with four three wire ports for dedicated servo motors.  The Propeller C3 runs on a 7.5-12V DC power supply which is much higher than the Arduino DUE, but this is the same voltage range that the motors would run on, which would allow us to be able to use one battery for the motors and the microcontroller without having to step down the voltage with a voltage regulator.

The Propeller C3 works as a full-fledged computer which gives it the advantage over the Arduino DUE in that it can run multiple high level languages instead of just proprietary software.  The other advantage it has is that it has dedicated servo wire ports which simplifies the wiring designs.  It also frees up other pins for sensor input and output.  It is slightly smaller and faster than the Arduino DUE.  The other advantage is that the Propeller C3’s architecture is a proprietary multicore processor.  The processor has eight “cogs” which can all run separate tasks.  These “cogs” have their own small memory caches and reach into a single pool for main memory.  This parallel architecture allows for faster processing in a parallel environment.  The operating system may be more flexible than the Arduino DUE, but it is not as flexible as a full-fledged computer.  The main drawback to the Propeller C3 is that it is much more expensive than the Arduino DUE.

Motion

In order for the robot to solve the maze it’s placed in, the robot must be able to move itself to the next square. The first step in the robotic design was to research and choose a robotic platform that would be best suited for the competition. There are a variety of platforms to choose from but the team decided which would best fit the robot.

Wheels

One of the most common platforms in robotics is wheels. Wheels are used throughout modern day life especially on cars, thus it is first choice for many due to its simplistic, familiar design. Some designs can range from using two wheels, four wheels, six wheels, or even more.  There is also a huge variety in choice. Tires come in many different sizes and treads to best suit whatever needs one may have. Wheels can also be a good economic choice since they are usually cheaper than many of the other available platform.

Treads

Another common platform used throughout the robotics industry is treads. Treads, much like wheels, are a great choice for its maneuverability and simple design. One of the advantages treads have over wheels is its ruggedness. Treads can easily overcome most obstacles it may meet. It has a great ability to climb over obstacles where wheels may have trouble. However there is a tradeoff. With this ruggedness comes the loss of speed, but since this is a small robotics competition this will not be an issue as it will be driving at relatively low speeds.

Swerve Drive

Swerve Drive is a system of wheels that have two separate motors per wheel. One motor drives the wheel forward and backwards while the other can turn the wheel on its side up to 90˚ to allow both forward and lateral movement without having to turn the entire robot. This allows for great maneuverability and a way to cut down on time wasted by turning. Since this a more complex system than most of the other choices, more time must be spent ensuring the system works properly. If one of the motors becomes misaligned from the others it can wreck the system resulting in a slower speed or it may travel in an unintended manner.  The cost of a swerve drive system is high due to the amount of motors needed for each wheel.

Mecanum

Mecanum wheels are similar to the swerve drive system in terms of maneuverability, in that they allow forward, backward, and lateral movement. The key difference is that Mecanum wheels do not need to rotate 90˚ for the lateral movement. Mecanum wheels are wheels with a series of rollers angled in a way to produce a lateral force. This results in a less complex system and greatly decreases time spent turning. One of the major issues with Mecanum wheels is the cost. The wheels themselves are expensive and can be priced over $100 for a set of four and a motor would be needed for each wheel. Another problem is the ruggedness of the wheels. Since the wheel consists of rollers, it can become unaligned with the intended path after it climbs an obstacle.

Battery Technology

To power the robot, a battery had to be chosen.  While this seems fairly straightforward, there were actually a good number of decisions that had to be made.  We started by looking at the various battery technologies available in the market and which would be the best fit for our robot.

Lead Acid

Lead acid batteries are an older technology, however still have a place in robotics.  Lead acid batteries have respectable discharge rates, usually somewhere in the window of 10C-20C.  This is lower than a few of the more modern technologies but is still very viable in today’s tech.  They also have decent capacities with the median capacity sitting around 4-10Ah.  However they have major downsides as well.  Lead acid batteries are extremely heavy and large, typically being larger than our entire robot would be and weighing on the upwards of 10-20 pounds.

Nickel Cadmium

NiCd batteries are also a fairly widespread option.  Their high potential discharge rates combined with stable/flat discharge characteristic makes them an attractive choice for our robot.  Many NiCd batteries are able to discharge at a rate up to 10C which is far more than enough current than we would need to run our motors.  However, NiCd batteries are extremely susceptible to the “memory effect”.  This effect is caused by the growths of larger crystals in the cell formation of the battery when the NiCd is attempted to be charged without being completely drained first.  Once the memory effect happens, the battery’s capacity can effectively be destroyed and is very difficult to reverse.

Nickel Metal Hydride

Ni-MH batteries are a more recent technology (1989) that don’t excel extremely well in most categories, yet don’t fail in any areas either.  NiMH batteries have a high energy density that are better than most technologies excluding Lithium battery types.  Most NiMH batteries have charge cycles in the thousands, usually around 3000.  They also don’t have many issues being used in a deeply discharged state (~80%+). Their discharge rates are also relatively high, yet not as high as other types such as Lead Acid or NiCd. The NiMH battery does suffer from a minute memory effect, yet this is noticeably not nearly as bad as in the NiCd battery.  They also deteriorate over time in storage rather quickly.  

Lithium Ion

Lithium Ion batteries are the frontrunners in consumer electronics’ batteries.  They have solved the major issue of the memory effect prevalent with NiMH and NiCd batteries, as well as most of the weight and size issues.  Lithium Ion batteries can be discharged at up to 40C or even higher in special optimized batteries, and can even have capacities on the upwards of 5-10Ah.  They have extremely high energy densities as well, even managing to easily beat out Lead Acid batteries.  Lithium Ion cells do have the issue of being slightly more expensive than most other battery types.  They also are very susceptible to losing capacity when overcharged and degrading if cell voltage ever drops below 2V (nominally 3.7V).

Lithium Polymer

Lithium Polymer batteries are an offshoot of regular lithium ion batteries, sometimes referenced as “battery in a sack” due to the housing of LiPo cells usually being a bag of some sort.  They are a more extreme version of Lithium Ion batteries.  They are even lighter and smaller than Li-Ion, flexible due to the lack of metal/rigid housing, and have even higher discharge rates.  However their energy density and life cycles are lower than Li-Ions.  A major downside to Polymers are the fact that if charged improperly, they can be damaged or even explode.

Alkaline

Out of all the other types we looked at, this is the one non-rechargeable option of the group.  Alkaline cells run at a higher cell voltage (1.5V) compared to most of the other rechargeable types that have a nominal voltage of 1.2V.  That means we would need to use less alkaline cells to reach our needed voltage of 12V.  However, alkaline batteries are not well suited to high current applications such as DC motors as they do not have a very high capacitance and cannot run for very long under such situations.  Alkaline cells would be well suited to running our microprocessors and sensors however.  Cost would become an issue as well once replacement costs are factored in over the course of our project.

Single Vs Multiple Batteries

The team next looked at how it should organize the power system for the robot.  One of the big decisions to make was whether to use a single battery or multiple batteries.  By using multiple batteries one could make sure each subsystem or module could have its own power source.  This would make it easy to match output voltages of batteries to exactly what each component needed.  However in doing this, it would quickly increase the cost along with weighing the robot down more and increasing complexity in organizing the wiring.  With one battery for the whole robot, the issue arises of running high current DC motors alongside lower voltage and current electronic components like microprocessors and cameras.

Voltage Regulation

With the combination of motors and electronic devices on our robot, it is necessary to create a way to distribute the correct voltage level to the correct modules.  Looking at all of the components on the robot, the DC motors run on 6V while all the electronics such as the sensors and controllers run on 5V.  This provided an easy solution of using voltage regulators to create 2 separate rails, one at 5V and another at 6V to power the whole robot from a single source.

After comparing the many different options available, the team ultimately decided on using 2 Hobbywing UBEC regulators.  These are preassembled circuits containing a voltage regulator IC, filter capacitors, and a selectable voltage output switch.  This specific model has a 3A throughput, and a selectable 5V/6V output.

By using these voltage regulators, two separate supply rails are created, isolating higher current and noisier motor power from the more sensitive control circuits.

Low Voltage Warning

The team determined that it would need an advance notice of when the robot’s battery is about to run out of power.  If the components are sent lower than rated voltages, there is a risk of damaging the electronics. The design is very simple and straightforward: Connecting 2 LEDs in parallel with an active zener diode and a resistive voltage divider.  A threshold voltage is selected, based on what voltage would trigger the warning, and appropriate resistor values are determined in order for that threshold voltage to correspond with 2.5V across the active zener diode.  Once 2.5V is hit, the zener will swap the flow of current into the LEDs, thus indicating that we need to charge the battery.

Algorithm

  1. Create an unfilled matrix for 5’x5’, 6’x6’, or 7’x7’ maze.
  1. Mark all squares around the grid as not part of the maze.
  1. Indicate on matrix where the predetermined start and finish squares are.
  2. Create a two-direction path from start to finish (up and left, up and right, down and right, or down and left) depending on where the finishing square is to the final square.
  3. Proceed to move to the next square on the two-direction path.
  4. Analyze walls around the robot.
  1. If the next square is blocked:
  1. Move to the second direction on the two-direction path.
  1. If both directions are blocked:
  1. Retrace steps until one of the two directions are not blocked and mark the path blocked.
  1. If the walls around the robot are colored blue or pink:
  1. Take an image of the wall and analyze it.
  1. If the next square is not blocked:
  1. Move to said square and mark square as part of the critical path.
  1. If the square is not blocked, but you have to back track
  1. Mark the square as not blocked, but not part of the critical path
  1. If the square you have reached is the ending square:
  1. Mark that you have finished the maze and wait for input.
  1. Update the square visited as blocked, part of the critical path, not part of the critical path, or not part of the maze.
  2. Repeat steps 4-6 until you have reached the final square
  1. Once you have reached the final square and have received the input from the judge, move through the maze only to squares you have marked as part of the critical path until you have reached the starting square.

Material Selection

Some materials that will be discussed in this report include the material for the treads and the base for the robot. These selections will be important as a wrong decision can cause the robot to fail or not work as efficiently as possible. Materials for other parts were not considered since these will be ordered from manufacturers.

Chassis

The chassis material must be strong enough to hold the weight of all the components of the robot and also be lightweight to reduce the strain on the motors. When researching, it was found that wood, aluminum, and acrylic plastic were the most widely used materials in making a robot chassis.

Aluminum was the first material considered as it is the strongest. Aluminum 6061 is the most common form of aluminum found in the US. With an ultimate yield strength of 45000 psi it easily outperforms the strength required for our design. The price of aluminum is $15.72 for a 1/8”x12”x12” sheet from onlinemetals.com. Since this is a metal, it will weigh a bit more than the other materials. With a density of .0975 lb/in3 it is one of the lightest metals but compared to the other options, it is still the heaviest. The other con is that it will conduct electricity. If the chassis comes into contact with any electrical circuit board, it will short out the board and leave our robot useless and possibly present a safety issue.

The second material researched was wood. This material is both lighter and cheaper than aluminum and comes in a variety of sizes and styles. It can easily be cut and shaped into what is required of the robot. It also does not conduct electricity which means electrical components can be secured directly to the chassis. Since the plywood in stores comes in layers, our design would have to change to make the chassis thicker to have a higher strength. While in most cases this would be a sound design choice, there is a size limitation that must be met.

The final material researched for the chassis design is acrylic plastic. With a yield tensile strength of 10000 psi, it can easily hold all the components without failing. It can be easily worked and cut to fit the robot design and is relatively cheap priced at $5.64 for a 1/8”x12”x12” sheet. Like the wood, all electrical components can be fastened directly to it since the material does not conduct electricity. With a specific gravity of 1.2, it is also lightweight and won’t strain the motors much.

Comparing all the materials together, it is easy to see that the acrylic plastic is the best choice for the robot due to its light weight, cheap price, and strength.

Safety

        There are some safety constraints that must be met in order to ensure that the robot is safe to operate.  If the robot is deemed unsafe to operate by the judges, it will be disqualified.

 The team’s design will require the use of many wires.  These wires will be run from sources of varying voltages.  In order to ensure that wires are not shorted, disconnected, or exposed, the wires should be tied down and secured.  They also should be tied away from any moving parts and away from the sides in order to avoid clipping by outside edges.  Exposed, disconnected, or shorted wires could lead to sparking, destruction of circuits, blown motor drivers, and burns from overheated components.

        To further ensure that our robot is safe to operate, edges must be rounded and smoothed in order to ensure safe handling.  Rough or sharp edges can penetrate the skin of the judges or the team as they handle the robot.  This could lead the judges deciding that the robot is not safe to operate, leading to disqualification.  

        The other safety constraint depends on the turning radius and speed of the robot.  During the competition, if the robot collides with a maze wall, points will be deducted from the final score.  Furthermore, if the robot causes any damage to the maze itself, The Team will be disqualified.  In order to ensure that the robot does not cause damage to the maze, The Team must establish a safe turning ratio in order to ensure that the robot will not unintentionally collide with the maze.  

        The team has also developed a low voltage warning system mentioned in the Power section.  The system will ensure that the voltage supply does not go below a certain threshold.  Going below this threshold without warning would cause the battery to discharge too far rendering it unchangeable and useless.  This could also lead to the robot running out of power at an unintended time.  Powering down prematurely could lead to the robot damaging itself.

Training a classifier

For this classification problem we have ten classes, the numbers 0 ... 9, so that our class labels are 0 ... 9. To train this ten-class classifier, we will use a dataset of images of cropped Sudoku cells. First we going to use a set of images for training. The images are named with the first character equal to the class (0 ... 9). The images are grayscale and roughly 80 ⇥ 80 pixels (with some variation). The following function takes an image and returns a feature vector of the flattened grayscale values.

def compute_feature(im):

""" Returns a feature vector for an

ocr image patch. """

# resize and remove border

norm_im = imresize(im,(30,30))

norm_im = norm_im[3:-3,3:-3]

return norm_im.flatten()

This function uses the resizing function imresize() from imtools to reduce the length of the feature vector. The picture also will be croped away about 10% border pixels since the crops often get parts of the grid lines on the edges.

Now we can read the training data using a function like this.

def load_ocr_data(path):

""" Return labels and ocr features for all images

in path. """

# create list of all files ending in .jpg

Extracting cells and recognizing characters

With a classifier that recognizes cell contents, the next step is to automatically find the cells. Once we solve that, we can crop them and pass the crops to the classifier. Let’s for now assume that the image of the Sudoku is aligned so that the horizontal and vertical lines of the grid are parallel to the image sides (like the left image of Figure 8.8). Under these conditions, we can threshold the image and sum up the pixel values horizontally and vertically. Since the edges will have values of one and the other parts values of zeros, this should give strong response at the edges and tells us where to crop. The following function takes a grayscale image and a direction and returns the ten edges for that direction.

from scipy.ndimage import measurements

def find_sudoku_edges(im,axis=0):

""" Finds the cell edges for an aligned sudoku image. """

# threshold and sum rows and columns

trim = 1*(im<128)

s = trim.sum(axis=axis)

# find center of strongest lines

s_labels,s_nbr = measurements.label(s>(0.5*max(s)))

m = measurements.center_of_mass(s,s_labels,range(1,s_nbr+1))

x = [int(x[0]) for x in m]

# if only the strong lines are detected add lines in between

if len(x)==4:

dx = diff(x)

x = [x[0],x[0]+dx[0]/3,x[0]+2*dx[0]/3,

x[1],x[1]+dx[1]/3,x[1]+2*dx[1]/3,

x[2],x[2]+dx[2]/3,x[2]+2*dx[2]/3,x[3]]

if len(x)==10:

return x

else:

raise RuntimeError(’Edges not detected.’)

A.13: Comparison Matrix and Analysis of Maze Solving Algorithm

To solve the maze we decided on the recursively maze solving algorithm.  

This table summarizes the characteristics of the Maze solving algorithms above. Maze solving algorithms can be classified and judged by these criteria. Descriptions of the columns follow:

  • Solutions: This describes the solutions the algorithm finds, and what the algorithm does when there's more than one. An algorithm can pick one solution, or leave multiple solutions. Also the solution(s) can be any path, or they can be the shortest path. The dead end and cul-de-sac fillers (and the blind alley sealer when considering its inaccessible sections) leave all solutions, however they may also leave passages that aren't on any solution path, so are marked "All +" above.
  • Guarantee: This is whether the algorithm is guaranteed to find at least one solution. Random mouse is "no" because it isn't guaranteed to terminate, and wall follower and Pledge algorithm are "no" because they will fail to find a solution if the goal is within an island. The dead end and cul-de-sac fillers are "no" because they may not do anything to the Maze at all in purely braid Mazes.
  • Focus: There are two general types of algorithms to solve a Maze: Focus on "you", or focus on the Maze. In a you-focuser, you have a single point ("You" above) or a set of points ("You +" above) and try to move them through the Maze from start to finish. In a Maze-focuser, you look at the Maze as a whole and invalidate useless passages.
  • Human Doable: This refers to whether a person could readily use the algorithm to solve the Maze, either while inside a life sized version, or while looking at a map from above. Some you-focuser algorithms can be implemented by a person inside (or above) the Maze, while some Maze-focusers can be implemented by a person, but only from above. Other algorithms are complicated or intricate enough they can only reliably be done by a computer.
  • Passage Free: This is whether the algorithm can be done anywhere. Some algorithms require the Maze to have obvious passages, or distinct edges between distinct vertices in graph terms, or one pixel wide passages when implemented on a computer. The wall follower, Pledge algorithm, and chain algorithm only require a wall on one side of you. The recursive backtracker and the shortest path(s) finders make their own paths through open spaces.
  • Memory Free: This is whether no extra memory or stack is required to implement the algorithm. Efficient algorithms only require and look at the Maze bitmap itself, and don't need to add markers to the Maze during the solving process.
  • Fast: This is whether the solving process is considered fast. The most efficient algorithms only need to look at each cell in the Maze once, or can skip sections altogether. Running time should be proportional to the size of the Maze, or in Computer Science terms O(n^2) where n is the number of cells along one side. Random mouse is slow because it isn't guaranteed to terminate, while the blind alley filler potentially solves the Maze from each junction.

A.16: Detailed Character Recognition Testing Plan

First we have to test basic functionality of the raspberry pi Camera and the raspberry Pi B+. The Pi has to be able to provide the camera with the sufficient power and read the right data sent from the camera to the GPU to for the camera to operate correctly and efficiently. Then we have to test if the camera can take pictures with the desired resolution and if the GPU can store them with relative square number using a predefined algorithm written by us that determines the location from the start position and the maze size. Then we have to check for that algorithm correctness. We also need to test the camera video capability as for how clear the taken videos, as that will help us to correctly determine the wall colors. The camera will also be calibrated with the other sensors after integration to see if they can response together to the robot position, (i.e. stopping at the intersections of the black lines in the middle of the squares) and the wall colors (i.e. taking picture) while the robot is not moving. Test also for picture storing in the Pi’s memory and if it can be recalled. Another test will be for the character identification accuracy with noise and distortion be applying noise to some stored pictures and/or taking pictures of the maze walls while robot is moving and test for how much recognition accuracy we can get. Test the robot for the ability of storing the output file for the competition judges, and the correct configuration.
